\section{Grundlagen Buffer Overflows}

\paragraph{a) Beschreiben Sie den Aufbau eines typischen Prozess-Stackframes und gehen Sie dabei besonders auf die verwendeten Register und deren Bedeutung ein. Ebenso gilt es zu beschreiben, was bei einem Funktionsaufruf im Detail auf dem Stackframe passiert. Dabei sind die Begriff Stack und Base-, Instruction Pointer zu verwenden.}~\\
\noindent Ein typischer Prozess-Stackframe besteht aus den übergebenen Parametern, einer Return-Adresse, dem gesicherten alten Base-Pointer und dem eigentlichen Stack und das von unten nach oben in dieser Reihenfolge. Die relevanten Register in der Struktur sind der Base- und Stack-Pointer. Der Base-Pointer zeigt dabei auf die Position des alten Base-Pointers und dient der Referenz auf die übergebenen Parameter. Der Stackpointer zeigt stets auf das oberste Element des Stacks. Die Return-Adresse ist der alte Instruction-Pointer des aufrufenden Programms.

Ruft ein Programm ein Unterprogramm auf, so werden stets gewisse Schritte vollzogen. Zuallererst werden die momentan vom aufrufenden Programm genutzten Register-Werte auf dem Stack gesichert. Als nächstes werden die übergebenen Parameter in fester Reihenfolge auf den Stack gepusht, dann der momentane Instruction-Pointer. Der Base-Pointer des aufrufenden Programms wird genauso gepusht und der Stackpointer wird zum aktuellen Basepointer. \footnote{$\rightarrow$ Dadurch zeigt der neue Base-Pointer auf den alten.} Werden ab jetzt lokale Variablen definiert, so landen diese oberhalb des Stacks. Auf diese wird per $[BP - x]$, auf die übergebenen Parameter wird per $[BP + x]$ referenziert. Ist das Unterprogramm fertig, wird der gesamte Stack abgebaut. Zeigt der Stack-Pointer auf den Base-Pointer wird per \textit{pop BP} der alte Base-Pointer rekonstruiert und per \textit{pop IP} der Instruction-Pointer.


\paragraph{b) Was ist ein Pointer? Gibt es Pointer in Java?}~\\
\noindent Ein Pointer ist ein Zeiger auf eine Adresse im Speicher. Im Gegensatz zu Variablen, in denen direkt Werte gespeichert werden, zeigt ein Pointer auf eine Adresse im Speicher. In dieser Adresse können dann Werte gespeichert sein. Beispielsweise können Pointer auf Objekte oder auf den Start einer Zeichenkette zeigen. Durch eine Dereferenzierung ist es dann möglich auf den Inhalt der Adresse zuzugreifen. In Java gibt es Pointer nicht direkt, also sie können nicht direkt von den Entwicklern verwendet werden, sondern werden versteckt, im Hintergrund von Java verwendet. Wenn in Java ein neues Objekt erzeugt wird, funktioniert dies im Hintergrund auch mit Pointern, auch wenn man als Entwicklerin nicht direkt die Pointer beeinflussen kann.

\paragraph{d) Welche Varianten von Buffer overflows gibt es? Warum sind diese nicht ohne weiteres in Java und ähnlichen Sprachen möglich? Wie kann man heutzutage Buffer overflows verhindern? Gehen Sie dabei auf hardware- und softwareseitige Maßnahmen ein.}~\\
\noindent Die beiden gängigsten und bekanntesten Buffer overflows sind der Stack und der Heap based Buffer overflow. Diese sind nicht direkt in Java möglich, weil es Programmiererinnen in Java nicht direkt möglich ist Pointer Arithmetik zu betreiben. Die Pointer und Buffer werden unter der Haube von Java verwendet und nicht direkt von den Programmiererinnen. Dadurch werden Programmierfehler, die zu Buffer overflows führen könnten verhindert. Es gibt verschiedene Möglichkeiten heutzutage Buffer overflows zu verhindern, z.B. gibt es in Hardware die Möglichkeite bestimmte Bereiche als nicht ausführbar zu markieren. Dadurch wird eingeschleuster Schadcode nicht ausgeführt. In Software gibt es die Möglichkeit einen Canary einzufügen. Dabei handelt es sich um einen zufälligen Wert, vor der Return Adresse steht. Wenn bei einem Buffer Overflow die Return Adresse überschrieben wird, dann wird zuvor der Canary überschrieben. Falls dieser also nicht mehr in Takt vorhanden ist, dann können Buffer Overflows damit erkannt werden.

\paragraph{e) Wozu verwendet man Shellcodes?}~\\
\noindent Shellcodes werden verwendet um eine Konsole zu öffnen. Als angreifende Person ist es möglich, darüber eine Shell z.B. mit Adminrechten zu öffnen.

\paragraph{f) Was ist NOP-Sliding?}~\\
\noindent Unter NOP-Sliding versteht man die Technik, bei einem Buffer Overflow mehrere NOPs vor dem Schadcode einzubauen. Dies macht man, da es bei Buffer Overflows häufig schwierig ist genau die richtige Adresse zu treffen, um das zu erleichtern, werden NOPs hinzugefügt, welche nichts bewirken. Zudem werden Systemabstürze verhindert, weil keine ungültigen Befehle ausgeführt werden.

\paragraph{g) Beschreiben Sie, wie Sie ein statisches und ein dynamisches array of char der Größe 100 in C erzeugen. Welcher Ansatz führt eher dazu, dass man einen Buffer Overflow anfälligen Code generiert.}~\\
\noindent Ein statisches Array kann mit folgender Zeile erzeugt werden:\\
char statischesArray[100];\\

Ein dynamisches Array wird so erzeugt:\\
ptr = (char*) malloc(100 * sizeof(char));\\

Bei einem statischen Array wird der Speicherplatz direkt auf dem Stack reserviert, Bei dem dynamischen Array wird der Speicherplatz hingegen auf dem Heap reserviert. Aus diesem Grund ist auch das statische Array anfälliger für Buffer Overflows, weil bei diesem Angriff häufig ausgenutzt wird, dass der Buffer auf dem Stack liegt und somit lokale Variablen und Rücksprungadressen überschrieben werden können.

\paragraph{i) Was bedeutet die Anweisung jmp esp? Wofür benötigt man diese zur Nutzung von Buffer Overflows? Recherchieren Sie, ob es eine Verwendung des JMP ESP Befehls gibt, die nicht mit der Verwendung bei Buffer overflows zusammenhängt. Tipp: betrachten Sie den Befehl in seiner Maschinendarstellung, sprich den Mnemonic Zahlenwert/Opcode.}~\\
\noindent Die Anweisung \textit{jmp esp} bedeutet, dass das Programm dahin springt, wo aktuell der Stack-Pointer hinzeigt. Häufig wissen Angreifer bei einem Buffer Overflow nicht genau die Adresse, wo der Schadcode letztendlich liegt. Um den Schadcode zur Ausführung zu bringen muss die Adresse des Schadcodes in den IP geschrieben werden. Dabei kann \textit{jmp esp} helfen. Es ist häufig einfach herauszufinden, wo ein Aufruf des Befehls \textit{jmp esp} liegt. Dann kann die return Adresse mit dieser Adresse überschrieben werden. Entsprechend wird nach dem Return, der \textit{jmp esp} Befehl ausgeführt. Nach dem Return wird auch der Stack-Pointer erhöht. Die Idee ist es, dass ein Angreifer Schadcode an die Stelle nach der Return-Adresse positioniert. Durch die Erhöhung des Stack-Pointers nach dem Return zeigt der Stack-Pointer auf den Schadcode. Wenn dann durch \textit{jmp esp} dahin gesprungen wird, wird entsprechend der Schadcode ausgeführt.

Anwendung von \textit{jmp esp} außerhalb von Buffer Overflows gibt es z.B. bei Just in Time Compilern, welche zur Laufzeit Maschinencode generieren. Um diesen direkt auszuführen, könnte \textit{jmp esp} verwendet werden.
